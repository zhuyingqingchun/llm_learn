# llm_learn 下一阶段方案报告（Qwen 路线）

## 1. 当前仓库状态概括

当前仓库已经具备两条较明确的学习与实验主线：

1. **从零实现 Transformer 翻译训练链路**：
   已有 `src/transformer/train.py`、`prepare_data.py`、推理与 DDP 辅助逻辑，能够完成数据处理、训练、验证、测试、TensorBoard 记录与检查点保存。

2. **GPT 风格预训练实验链路**：
   已有 `src/gpt/cli.py` 与 `makefile` 中的 `gpt-offline` 入口，说明仓库已经开始从“教学型 Transformer”向“语言模型训练工程化”过渡。

3. **项目已经是 Python 包化结构**：
   `pyproject.toml` 中已经定义了脚本入口，但整体仍偏“学习代码集合”，尚未进入“面向 Qwen 的标准化微调项目”阶段。

因此，下一阶段不建议继续横向堆零散实验，而应正式进入：

> **“以 Qwen 为主线，把仓库从学习型实验仓库，推进到可复用的大模型微调与评测仓库。”**

---

## 2. 下一阶段的总目标

下一阶段建议只做一件主线工作：

**建立 Qwen 微调/评测/推理的最小可复用工程骨架。**

这里的重点不是立刻追求复杂算法，而是先补齐以下四个缺口：

- **模型入口统一**：把 `Qwen2.5-1.5B / 7B / Qwen3-Next-80B-A3B-FP8` 的用途分清。
- **数据入口统一**：建立后续 SFT / DPO / 蒸馏都能复用的数据格式。
- **训练入口统一**：先支持 Qwen SFT，再为 DPO 留接口。
- **评测入口统一**：至少形成 `base -> sft -> dpo` 的可对照评测闭环。

---

## 3. 模型分工建议

### 3.1 Qwen2.5-1.5B-Instruct
定位为：

- 流程验证模型
- 小样本调参模型
- 后续蒸馏学生模型

不承担主结果产出任务，主要负责低成本试错。

### 3.2 Qwen2.5-7B-Instruct
定位为：

- 主训练模型
- 主评测模型
- 第一阶段核心基座模型

下一阶段大多数时间都应围绕这个模型展开。

### 3.3 Qwen3-Next-80B-A3B-Instruct-FP8
定位为：

- 教师模型
- 高质量推理模型
- 偏好数据/蒸馏数据生成器

现阶段不建议把它作为主训练对象；它更适合后续 serving 与数据反哺。

---

## 4. 仓库结构的下一步方向

建议下一阶段不要打散在现有 `src/transformer` 和 `src/gpt` 里继续堆逻辑，而是新增一条独立路线：

```text
src/
  qwen/
    config.py
    dataset.py
    sft_train.py
    dpo_train.py
    infer.py
    evaluate.py
```

其中：

- `config.py`：集中放训练、模型、数据参数
- `dataset.py`：统一 messages/jsonl 数据读取与格式转换
- `sft_train.py`：第一阶段主入口
- `dpo_train.py`：第二阶段入口，先预留
- `infer.py`：单条推理/批量推理
- `evaluate.py`：统一评测接口

这样做的原因是：

1. 不污染你已有的“手写 Transformer / GPT 学习代码”
2. Qwen 路线与从零实现路线职责清晰
3. 后续你要切到 LoRA / QLoRA / DPO / vLLM 时不需要重构整个仓库

---

## 5. 数据方案建议

下一阶段的数据策略不应是“先下很多数据集”，而应是“先统一格式，再逐步扩数据”。

建议统一成一种基础结构，例如：

```json
{
  "id": "sample-0001",
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "请解释一下 Transformer 的自注意力机制。"},
    {"role": "assistant", "content": "..."}
  ],
  "source": "manual"
}
```

下一阶段数据只分三类：

### A. 流程验证数据
用于跑通训练链路。

### B. 主训练数据
用于 7B 的 SFT。

### C. 偏好/蒸馏数据
用于 DPO 和教师模型反哺。

重点不是量大，而是：

- 字段统一
- 能复用
- 后续易于自动清洗与统计

---

## 6. 训练路线建议

下一阶段建议固定为：

```text
阶段 1：Qwen2.5-1.5B 跑通 SFT 流程
阶段 2：Qwen2.5-7B 做主 SFT
阶段 3：Qwen2.5-7B 接 DPO
阶段 4：Qwen3-Next-80B-A3B-FP8 做教师采样/蒸馏
```

这里故意不建议一开始就上复杂 RL，原因是：

- 你当前仓库还没有 Qwen 训练骨架
- 数据集还没形成稳定规范
- 评测闭环也还没标准化

所以最优顺序是：

> **先工程化，再算法化；先 SFT，再 DPO；最后再碰更复杂的强化学习路线。**

---

## 7. 评测建议

下一阶段必须补上评测，而不是只看 loss。

建议最低配做四类评测：

1. **基础指令跟随**
2. **数学/推理**
3. **你的自定义任务集**
4. **长文本样例**

评测结果建议统一输出到：

```text
artifacts/
  eval/
    base/
    sft/
    dpo/
```

并生成简单的 markdown 或 json 汇总，方便后续比较。

## 10. 最终结论

结合当前仓库状态，下一阶段最合理的方向不是继续扩展“手写模型细节”，而是：

> **以 Qwen 为核心，正式把仓库升级成一个可复用的大模型微调实验仓库。**

这个方向能最大化保留你当前仓库已有的学习价值，同时把后续工作真正推进到“大模型优化与工程化”主线上。
